Overview

BAC4 Integration Modeller provides a JSON Schema-based, standards-compliant approach to defining, validating, and generating integration implementations. It adopts industry standards like AsyncAPI, OpenAPI, and CloudEvents while offering extensibility through plugins.

Create Your First Integration

# Initialize a new integration definition

Key Insights

ğŸ’¡ No Unified Standard
While AsyncAPI, OpenAPI, and CloudEvents exist for specific use cases, there is no single unified standard for defining all integration patterns (REST, Kafka, MQ, File Transfer, Batch, etc.) in one cohesive format.
âœ… Opportunity Identified
BAC4 Integration Modeller fills this gap by providing a unified JSON Schema that supports multiple integration patterns while adopting existing standards where applicable.

ğŸ“ Standards Analysis

AsyncAPI Specification

Version: 3.0.0 (latest), 2.x series widely adopted

Purpose: Machine-readable definitions of asynchronous/event-driven APIs

Format: JSON or YAML

Key Features:

  • 100% compatible with JSON Schema Draft 07
  • Protocol-agnostic (supports Kafka, MQTT, AMQP, WebSocket, etc.)
  • Supports channels, messages, schemas, and operations
  • Extensive tooling ecosystem for validation, documentation, and code generation

Use Cases:

  • Event streaming (Kafka topics)
  • Message queues (RabbitMQ, ActiveMQ)
  • WebSocket APIs
  • MQTT for IoT

Adoption:

Widely used by major cloud providers and enterprises for event-driven architectures.

OpenAPI Specification (OAS)

Version: 3.1.0 (latest)

Purpose: Standard interface description for RESTful APIs

Format: JSON or YAML

Key Features:

  • Language-agnostic API definitions
  • Machine and human-readable
  • Supports paths, operations, parameters, request/response schemas
  • Extensive tooling for validation, mocking, testing, code generation

Use Cases:

  • REST APIs
  • HTTP-based integrations
  • API documentation (Swagger UI)
  • Contract-first API development

Adoption:

Industry standard overseen by the OpenAPI Initiative (Linux Foundation project).

CloudEvents Specification

Version: 1.0

Status: CNCF Graduated Project (January 2024)

Purpose: Common format for describing event data

Format: JSON (with multiple protocol bindings)

Key Features:

  • Standardized event envelope
  • Multiple protocol bindings (HTTP, Kafka, AMQP, MQTT)
  • Core attributes: id, source, type, datacontenttype, time, etc.
  • Improves interoperability across platforms

Use Cases:

  • Event-driven architectures
  • Cross-platform event exchange
  • Serverless computing
  • Distributed systems

Adoption:

AWS EventBridge, Azure Event Grid, Google Cloud Eventarc, SAP event services.

BAC4 Standards Adoption Strategy

âœ… Hybrid Approach
BAC4 Integration Modeller adopts a hybrid approach:
  • Core unified JSON Schema for all integration patterns
  • Embed/reference AsyncAPI for event-driven integrations
  • Embed/reference OpenAPI for REST API integrations
  • Use CloudEvents format for event envelopes in Kafka/messaging patterns
  • Extend with custom schemas for patterns not covered by existing standards

Kafka vs Traditional MQ

Kafka Advantages
  • High throughput (millions/sec)
  • Message retention and replay
  • Horizontal scalability
  • Pull model (consumer control)
  • Stream processing built-in
MQ Advantages
  • Complex routing patterns
  • Transaction support (XA)
  • Message acknowledgment
  • Priority queues
  • Request-reply patterns

When to Use Each Pattern

Use REST API when:
  • Synchronous request-response needed
  • External APIs or third-party integrations
  • CRUD operations on resources
  • Human-readable, widely supported standard
Use Kafka when:
  • High-throughput event streaming required
  • Real-time data pipelines
  • Event sourcing architecture
  • Multiple consumers need same data
  • Message replay capability needed
Use MQ when:
  • Guaranteed message delivery required
  • Transaction support needed
  • Complex routing logic
  • Point-to-point communication
  • Legacy system integration
Use Batch when:
  • Large volume data processing
  • Scheduled/periodic execution
  • Complex data transformations
  • Data warehousing/ETL
  • Real-time not required
Use File Transfer when:
  • Large files (GB+)
  • B2B partner integrations
  • Legacy system constraints
  • Regulatory compliance (EDI)
  • Bandwidth-optimized transfer
